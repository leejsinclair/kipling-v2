# Kipling Story Plan 2 - LLM Integration for Iterative Scoring

## 1. Purpose
Add an AI-assisted review loop to Kipling that works with the existing scoring engine instead of replacing it.

Implementation reference:
- Review `PROJECT_STRUCTURE.md` before coding to understand current file layout, state flow, scoring modules, and test locations.

The new flow should support:
- Story draft scoring
- AI suggestions and optional rewrite
- User confirmation of accepted changes
- Final story scoring
- The same draft -> improve -> confirm -> final process for acceptance criteria

This keeps users in control while making improvement steps explicit and measurable.

---

## 2. Product Fit with Current Kipling
Kipling currently has a phase-based flow:
- `story` phase: user writes and scores the story
- `criteria` phase: user writes and scores acceptance criteria
- `complete` phase: combined result and export

This plan extends that model with review checkpoints:
- `story-draft` -> `story-review` -> `story-confirmed`
- `criteria-draft` -> `criteria-review` -> `criteria-confirmed`

Final XP and history persistence happen only after confirmation and final scoring.

---

## 3. User Stories
- As a user, I want a draft score before AI suggestions so I can understand my baseline.
- As a user, I want AI to suggest improvements without auto-overwriting my text so I stay in control.
- As a user, I want to confirm story changes before the final score is locked.
- As a user, I want the same confirmation process for acceptance criteria.
- As a user, I want to compare draft and final scores so I can see measurable improvement.

---

## 4. End-to-End Workflow

### 4.1 Story Workflow
1. User writes story (`As a`, `I want`, `So that`).
2. User clicks `Score Draft Story`.
3. Kipling scoring engine returns a draft score and breakdown.
4. User clicks `Improve with AI` (optional).
5. LLM returns:
   - issues found
   - suggested rewrite
   - rationale mapped to Kipling scoring criteria
6. User chooses:
   - accept full suggestion
   - accept partial edits
   - reject and keep own version
7. User clicks `Confirm Story`.
8. System computes final story score from confirmed text.

### 4.2 Acceptance Criteria Workflow
1. User enters criteria (Gherkin or bullet format).
2. User clicks `Score Draft Criteria`.
3. Criteria engine returns draft score and breakdown.
4. User clicks `Improve Criteria with AI` (optional).
5. LLM returns:
   - criterion-level improvements
   - format corrections
   - testability/clarity guidance
6. User confirms accepted criteria.
7. System computes final criteria score from confirmed criteria.

### 4.3 Completion
- Combined score is calculated from final story + final criteria scores.
- XP is awarded from final scores (not draft scores).
- Session history stores both draft and final values for transparency.

---

## 5. Functional Requirements

### 5.1 Draft vs Final States
The system must keep separate values for:
- story draft text and score
- story confirmed text and final score
- criteria draft text and score
- criteria confirmed text and final score

### 5.2 AI Improvement Controls
Users can:
- request AI suggestions after draft scoring
- preview AI suggestions before applying
- accept all, accept some, or reject suggestions
- re-score after edits

### 5.3 Confirmation Gate
The system must require explicit user confirmation before:
- final scoring
- XP updates
- saving completed entry in history as final

### 5.4 Explainability
AI responses must include:
- what changed
- why it improves score quality
- which scoring dimension the change targets

### 5.5 Provider and API Key
- User provides their own OpenAI API key at runtime.
- The key is used only for the active browser session (in-memory state only).
- The key must not be written to localStorage, sessionStorage, cookies, URL params, logs, or history records.
- If the tab is refreshed or closed, the key is cleared and must be re-entered.
- LLM calls should fail gracefully with clear user feedback.

### 5.6 AI Feature Gating
- AI-powered controls are disabled by default.
- When no valid API key is present, AI controls show a clear disabled state and helper text.
- After a valid key is provided, AI controls are enabled immediately for the current session.
- If key validation fails or the key is revoked/expired, AI controls return to disabled state.

## 8. Technical Flow

```text
User Enters API Key
-> In-Memory Session State
-> Validate Key
-> Enable AI Controls

Story Draft Input
-> Draft Story Score (local engine)
-> AI Suggestion Request (optional)
-> User Accept/Reject Edits
-> Story Confirmation
-> Final Story Score

Criteria Draft Input
-> Draft Criteria Score (local engine)
-> AI Suggestion Request (optional)
-> User Accept/Reject Edits
-> Criteria Confirmation
-> Final Criteria Score

Final Story Score + Final Criteria Score
-> Combined Score + XP + History Save

Session Ends / Tab Refresh
-> Clear In-Memory API Key
-> Disable AI Controls
```

## 9. Acceptance Criteria

### Story Draft Scoring
Given a user writes a story
When they click `Score Draft Story`
Then the system shows draft story score and breakdown without finalizing progress.

### AI Access Gating
Given no valid API key is present
When the user views AI controls
Then AI controls are disabled and the UI explains that a session key is required.

Given a valid API key is provided
When validation succeeds
Then AI controls are enabled for the current session only.

Given the browser tab is refreshed or closed
When a new session starts
Then the API key is not present and AI controls are disabled until a key is re-entered.

### Story Improvement and Confirmation
Given a draft story score exists
When the user requests AI improvement
Then the system shows suggested edits and rationale mapped to scoring rules.

Given suggestions are shown
When the user confirms the final story text
Then the system calculates and stores a final story score from confirmed content.

### Criteria Draft Scoring
Given criteria are entered
When the user clicks `Score Draft Criteria`
Then the system shows draft criteria score and breakdown without finalizing progress.

### Criteria Improvement and Confirmation
Given a draft criteria score exists
When the user requests AI improvement
Then the system returns criterion-level suggestions and format guidance.

Given suggestions are shown
When the user confirms final criteria
Then the system calculates and stores a final criteria score from confirmed criteria.

### Finalization
Given final story and criteria scores exist
When the session is completed
Then the system calculates combined score, awards XP, and saves session history using final values.

## 10. Non-Functional Requirements
- Keep scoring deterministic in local engines; AI augments wording and structure.
- Keep draft/final transitions fast and understandable.
- Preserve offline-safe behavior where possible; AI path is optional.
- Ensure prompts avoid exposing private user data beyond required story content.
- Treat API keys as sensitive runtime secrets and avoid client-side persistence.

---

- Story flow supports draft score -> AI improve -> confirm -> final score.
- Criteria flow supports draft score -> AI improve -> confirm -> final score.
- Combined score uses final values only.
- XP and history persistence happen only after final confirmation.
- UI clearly displays draft vs final scores and deltas.
- AI output includes explanation tied to Kipling scoring dimensions.

## 13. Integration Testing Without a Real OpenAI Key

### 13.1 Goal
Enable full integration-style testing of AI workflows in CI and Copilot Cloud without requiring a working OpenAI API key.

### 13.2 Testing Strategy
- Use a provider abstraction in `src/llmService.js`.
- Add a deterministic `mock` provider path for tests.
- Keep all app-level tests focused on behavior (UI state, flow transitions, scoring updates), not external network success.

### 13.3 Provider Contract for Tests
`src/llmService.js` should expose stable methods:
- `validateOpenAIKey(apiKey)`
- `improveStoryWithAI({ story, draftScore, breakdown, apiKey })`
- `improveCriteriaWithAI({ criteria, format, draftScore, breakdown, apiKey })`

Test mode behavior:
- Accept `test-key` as valid.
- Reject obvious invalid values with deterministic `invalid key` error.
- Return canned improvement payloads for story and criteria.
- Provide deterministic failures for `quota` and `network` simulation.

- `src/test/fixtures/llmResponses.js`
